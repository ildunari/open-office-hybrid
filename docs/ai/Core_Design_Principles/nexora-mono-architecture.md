# NexoraSIM™ — Architecture & Implementation Design

## Overview

NexoraSIM is a marketing/product landing page for an AI-driven eSIM and IoT connectivity platform targeting Myanmar's telecommunications market. It's a single-page application that presents the platform's capabilities through five scrollable sections: a hero with an animated globe visualization, core eSIM features (ICCID search, SM-DP+ integration, QR activation), an "OneX™ AI Engine" showcase with a live neural network diagram, Vercel infrastructure details with a simulated deployment pipeline, and a footer with system status and contact information. The site is purely presentational — no backend, no authentication, no data fetching — with all "live metrics" simulated client-side via `setInterval`.

### Rendering Strategy

The entire app is a **single route** (`/`). The root `page.tsx` is a React Server Component that composes six child components. All children are marked `"use client"` because they use `useState`, `useEffect`, and DOM APIs (canvas, scroll listeners, timers). The `<Suspense>` boundary wraps all sections below the nav with `<LoadingSpinner>` as fallback.

There is no SSR data fetching, no API routes, no `getServerSideProps` or `generateStaticParams`. The page is effectively a **static shell** that hydrates into a fully client-driven SPA with animated visualizations.

### Data Flow

There is no external data. All content is hardcoded in component files as:
- Object arrays (`features`, `panels`, `vercelFeatures`, `navItems`)
- State variables initialized to fixed values, then mutated by `setInterval` timers to simulate "live" metrics

**State lifecycle per component:**

1. **Initial render** — static content displayed
2. **`useEffect` on mount** — starts multiple `setInterval` timers that update state every 2–15 seconds
3. **Re-renders** — triggered by timer-driven `setState` calls (metric jitter, node activation, deployment status cycling)
4. **Cleanup** — all intervals cleared on unmount

### State Management

No global state management library. Each component manages its own local state via `useState`:

| Component | State Variables | Purpose |
|-----------|----------------|---------|
| `Hero` | `rotation`, `pulseScale`, `connectionNodes` | Globe rotation angle, breathing scale, node activation |
| `Features` | `activeFeature`, `connectionLines` | Selected card index, inter-card connection animation |
| `AISection` | `activePanel`, `processingData`, `metrics` | Selected panel, processing overlay toggle, metric jitter |
| `VercelSection` | `deploymentStatus`, `edgeMetrics` | Status cycle (READY→BUILDING→DEPLOYING→LIVE), metric jitter |
| `Navigation` | `scrolled`, `activeSection`, `mobileMenuOpen`, `showLeftScroll`, `showRightScroll` | Scroll state, active section from scroll spy, mobile menu, horizontal scroll arrows |
| `Footer` | `systemStatus` | Random status text cycling |

### Component Details

**`Navigation`** — Fixed-position top bar. Implements scroll spy via `window.scroll` listener to highlight active section. Horizontally scrollable nav links with chevron buttons when overflow is detected. Mobile hamburger menu with animated dropdown. Props: none. Key refs: `navScrollRef` for horizontal scroll container.

**`Hero`** — Full-viewport hero section with three visual layers:
1. A `<canvas>` element with 30 floating particles animated via `requestAnimationFrame`
2. An inline SVG globe (384×384) with concentric circles, meridian lines, and animated connection nodes. Rotates continuously via `setInterval` updating a CSS `rotate()` transform.
3. Headline text, status indicators, and KPI stats grid.

Props: none. Key behavior: dual animation loop (canvas rAF + React state interval).

**`Features`** — Three feature cards in a responsive grid. Each card has corner bracket decorations that expand on hover, an inline SVG schematic diagram, and expandable metrics revealed when the card is active. Auto-cycles active card every 5s. SVG connection lines between cards animate randomly.

Props: none. Uses Lucide icons: `Search`, `Cpu`, `QrCode`.

**`AISection`** — Two-column layout. Left: three accordion-style panels (Workflow Automation, Predictive Analytics, Smart Provisioning) with click-to-expand revealing metrics and process flow steps. Right: a 4-layer neural network SVG visualization with pulsing nodes during simulated processing, plus a 2×2 metrics grid with jittering values.

Props: none. Uses Lucide icons: `Brain`, `Zap`, `Settings`.

**`VercelSection`** — Inverted color section (black background, white text). Contains: deployment status bar cycling through READY→BUILDING→DEPLOYING→LIVE, a 4-column feature card grid, and a vertical deployment architecture diagram (CLI → EDGE/CDN/FN → APP) with state-aware indicator dots.

Props: none. Uses Lucide icons: `Cloud`, `Zap`, `Eye`.

**`Footer`** — System status bar, 4-column grid (brand info + GSMA cert, contact details, performance metrics, and footer links). Decorative grid pattern and geometric shapes. Copyright and technology badges at bottom.

Props: none. Uses Lucide icons: `Mail`, `Phone`, `MapPin`.

**`LoadingSpinner`** — Simple full-screen spinner with rotating border circle and "LOADING NEXORASIM..." text. Server component (no `"use client"` directive).

## Data Model

There are no TypeScript interfaces or types defined beyond inline component state. All data structures are anonymous object literals:

```typescript

{ x: number; y: number; active: boolean }[]

// Hero — canvas particles
{ x: number; y: number; vx: number; vy: number; opacity: number }[]

// Features — feature definition

  title: string;
  description: string;

  metrics: Record<string, string>;
  status: string;         // "ACTIVE" | "OPERATIONAL" | "READY"
  icon: LucideIcon;
  schematic: JSX.Element; // inline SVG
}

// Features — connection lines
{ from: number; to: number; active: boolean }[]

// AISection — panel definition
{
  title: string;
  description: string;
  metrics: string[];     // display labels
  processes: string[];   // flow step names
  icon: LucideIcon;
}

// AISection — live metrics
{ throughput: number; accuracy: number; latency: number; efficiency: number }

// VercelSection — feature definition
{
  title: string;
  description: string;
  status: string;
  metrics: Record<string, string>;
  icon: LucideIcon;
}

// VercelSection — edge metrics
{ regions: number; requests: number; latency: number; uptime: number }

// Navigation — nav item
{ id: string; label: string }

// Footer — system status
{ api: string; database: string; network: string; security: string }
```

All data is hardcoded. No APIs, no database, no CMS, no static data files.

## Routing & Navigation

**Single route:** `/` renders `app/page.tsx`.

**In-page navigation** via anchor links (`#hero`, `#features`, `#ai`, `#vercel`, `#contact`). The `Navigation` component implements scroll spy to detect the active section and applies smooth scrolling via `html { scroll-behavior: smooth }`.

Nav items also include `#docs`, `#api`, `#support` which have no corresponding sections — they're dead links, likely placeholders for future pages.

**No additional routes**, no dynamic routes, no API routes, no middleware.

## Patterns & Conventions

### Naming

- **Files:** PascalCase for components (`AISection.tsx`), kebab-case for shadcn/ui (`button.tsx`, `alert-dialog.tsx`), kebab-case for hooks (`use-mobile.ts`)
- **Components:** PascalCase, default exports for page components, named exports for UI primitives
- **CSS classes:** Tailwind utilities exclusively in JSX; custom classes in `globals.css` use kebab-case (`animate-fadeIn`, `hover-lift`, `circuit-pattern`)
- **Variables:** camelCase, descriptive (`connectionNodes`, `deploymentStatus`, `edgeMetrics`)

### Import Aliases

- `@/*` maps to project root (configured in `tsconfig.json`)
- Used in `components/ui/*` files: `@/lib/utils`, `@/components/ui/*`
- App components use relative imports (`./components/Hero`)

### Component Patterns

- **CVA variant system** for all shadcn/ui components (button, badge). Variants defined with `cva()`, props extend HTML attributes + `VariantProps`.
- **`asChild` pattern** via `@radix-ui/react-slot` for polymorphic rendering (button can render as anchor, etc.)
- **`cn()` utility** (`lib/utils.ts`) chains `clsx` → `tailwind-merge` for all className composition.
- **forwardRef** used consistently in shadcn/ui primitives.
- **No custom hooks** used by page components — only `useState`, `useEffect`, `useRef` from React.

### Animation Strategy

Heavy use of animations across three techniques:
1. **CSS animations** — `@keyframes` in globals.css (`fadeIn`, `slideInFromLeft`, `slideInFromRight`, `slideDown`, `dataFlow`)
2. **Tailwind animation utilities** — `animate-pulse`, `animate-bounce`, `animate-ping`, `animate-spin` + custom `animate-spin-slow`, `animate-pulse-slow`
3. **JavaScript-driven** — `setInterval` for React state updates, `requestAnimationFrame` for canvas particle system, inline SVG `<animate>` elements

### Accessibility

- `prefers-reduced-motion` media query suppresses all animations
- `prefers-contrast: high` increases contrast on key elements
- `aria-label` on mobile menu toggle and scroll buttons
- Print stylesheet hides `.no-print` elements
- `font-display: swap` on web fonts
- Missing: skip-to-content link, ARIA landmarks on sections, focus management in mobile menu

### SVG Patterns

Every section uses inline SVG for either decorative backgrounds (circuit board patterns) or data visualizations (globe, schematics, neural network, deployment architecture). SVGs are defined inline in JSX, not imported as files.
